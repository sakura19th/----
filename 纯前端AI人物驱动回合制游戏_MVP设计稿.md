# 纯前端 AI 人物驱动回合制游戏 MVP 设计稿

## 1. 项目概述

### 1.1 项目名称
**无限世界**（暂定）

### 1.2 一句话定义
一个以 **AI 生成人物与事件** 为特色、以 **小队回合制战斗** 为核心、以 **单局冒险** 为结构的纯前端网页游戏。

### 1.3 MVP 目标
在尽量低的开发复杂度下，做出一个 **可以完整游玩一局** 的版本，并让玩家明确感受到：

1. 每局遇到的人物不同
2. 角色有性格、有关系、有立场
3. 战斗系统本身具备可玩性
4. AI 让内容更鲜活，但不会破坏规则稳定性

---

## 2. 设计原则

### 2.1 核心原则
- **程序定义规则，AI 生成内容**
- **角色驱动优先，剧情辅助推进**
- **先保证可玩，再追求无限变化**
- **先做单局闭环，再做长线成长**

### 2.2 AI 的职责边界
AI 负责：
- 角色背景
- 角色性格标签
- 台词风格
- 支线事件文本
- 剧情节点描述
- 结局总结

AI 不负责：
- 战斗结算
- 伤害计算
- Buff / Debuff 逻辑
- 掉落概率
- 数值平衡
- 敌人行为底层规则

### 2.3 MVP 成功标准
满足以下条件即视为 MVP 成功：
- 玩家可以从开局打到结局
- 单局时长 30~60 分钟
- 至少 3 名可入队角色
- 至少 6~8 个事件节点
- 至少 5 场战斗
- AI 生成的人物与事件可正常接入流程
- 即使 AI 请求失败，游戏仍可用模板继续进行

---

## 3. 核心体验定义

### 3.1 玩家体验目标
玩家在一局中应体验到：
- 招募不同性格的伙伴
- 在事件中做选择
- 感受到角色对选择的不同态度
- 通过战斗构筑一个有协同的小队
- 在结局看到一段“属于这局”的冒险总结

### 3.2 关键词
- 人物驱动
- 小队构筑
- 回合制战斗
- 节点推进
- 单局 roguelite
- AI 叙事增强

### 3.3 非目标
MVP 不做：
- 开放世界
- 实时战斗
- 多结局超长主线
- 联机
- 大地图自由探索
- 复杂装备锻造
- PVP
- 过量数值养成

---

## 4. 世界观与题材方向

### 4.1 推荐题材
为降低美术与叙事成本，MVP 推荐采用：

**奇幻废土 / 神秘都市 / 异界旅团** 之一。

推荐最终选择：

### 4.2 MVP 世界设定提案
**题材：异界旅团 + 碎片世界**

世界被分裂成许多漂浮的区域，旧文明崩坏后，不同阵营在废墟、教团、商会、遗迹之间争夺“世界核心碎片”。玩家扮演一名旅团引路人，在旅途中招募不同性格与立场的同伴，经历事件、战斗与抉择，最终面对一场关键冲突。

这个设定的优点：
- 适合节点推进
- 容易合理化“每局不同人物”
- 容易解释“AI 生成角色来自不同区域与背景”
- 适合做单局叙事

---

## 5. 游戏循环（Core Loop）

### 5.1 单局主循环
1. 开局生成本局种子与背景摘要
2. 玩家选择初始主角原型
3. 进入节点地图
4. 在事件 / 战斗 / 招募 / 营地 / 商店节点间推进
5. 逐步组建最多 3 人队伍
6. 面对精英与 Boss
7. 结局结算，生成冒险编年史

### 5.2 单节点循环
1. 进入节点
2. 展示节点描述
3. 玩家进行选择
4. 更新队伍 / 资源 / 关系 / 事件状态
5. 如触发战斗则进入战斗界面
6. 战斗结束后回到地图

### 5.3 单场战斗循环
1. 初始化双方单位与状态
2. 根据速度决定行动顺序
3. 玩家选择行动
4. 执行动作与结算状态
5. 敌人行动
6. 继续至一方胜负决定
7. 发放奖励并返回节点流程

---

## 6. MVP 玩法范围

### 6.1 单局结构
- 总节点数：8~12
- 普通战斗：3~5 场
- 精英战斗：1 场
- Boss 战：1 场
- 招募节点：2~3 个
- 剧情事件：3~4 个
- 营地/恢复节点：1~2 个
- 商店节点：1 个

### 6.2 队伍规模
- 玩家小队最大人数：3
- 上场人数：3
- 替补系统：MVP 不做

### 6.3 资源类型
MVP 只保留最小必要资源：
- **生命值 HP**
- **能量/技力 SP**
- **碎晶（货币）**
- **关系倾向（隐藏或半隐藏）**

### 6.4 成长方式
MVP 成长方式：
- 战斗后获得少量属性成长或临时强化
- 事件中获得技能升级 / 被动加成 / 道具
- 招募新角色形成阵容协同

不做复杂等级树，只做轻成长。

---

## 7. 人物驱动系统设计

### 7.1 角色在 MVP 中的地位
角色是整个游戏的核心资产。剧情不是主角，角色才是主角。

玩家主要记住的应该是：
- 我这局招募了谁
- 他们怎么看我
- 他们之间关系如何
- 他们在战斗里好不好用

### 7.2 角色构成
每个角色由两部分组成：

#### A. 规则层（程序定义）
- 职业
- 属性
- 基础数值
- 技能模板
- 战斗定位
- 稀有度（MVP 可选）

#### B. 表现层（AI 或模板生成）
- 名字
- 称号
- 背景故事
- 性格标签
- 说话风格
- 对主角的初始态度
- 对其他角色的关系描述

### 7.3 角色原型池
MVP 推荐先做 8~12 个角色原型模板，再用 AI 在其上做变化。

示例原型：
- 流亡骑士
- 黑市医师
- 遗迹学者
- 教团叛徒
- 赏金猎人
- 机械修补师
- 沉默守卫
- 失忆术士

每个原型固定：
- 职业定位
- 技能模板范围
- 性格倾向池
- 关系冲突偏好

### 7.4 性格标签系统
每个角色拥有 2~3 个性格标签，例如：
- 冷静
- 冲动
- 务实
- 理想主义
- 傲慢
- 善良
- 多疑
- 忠诚
- 贪婪
- 悲观

作用：
- 决定台词风格
- 影响事件反应
- 影响角色关系
- 与战斗风格做轻绑定

### 7.5 关系系统（MVP 简化版）
MVP 不做复杂社交模拟，只做轻量关系值：
- 支持
- 中立
- 反感

关系变化来源：
- 玩家选择
- 特定事件
- 角色共同出战
- 某些剧情节点

关系的效果：
- 解锁特殊台词
- 影响结局总结
- 在战斗中触发简单连携文案

## 8. 战斗系统设计

### 8.1 战斗目标
战斗系统必须在没有 AI 文本加持时依然成立。

### 8.2 战斗形式
- 3 人小队 vs 1~3 名敌人
- 严格回合制
- 按速度排序行动
- 玩家每回合从行动菜单中选择操作

### 8.3 可选行动
每回合角色可执行：
- 普攻
- 技能 1
- 技能 2
- 防御
- 使用道具（MVP 可选）

### 8.4 核心战斗机制
MVP 推荐只实现 5 种机制：
- 护盾
- 中毒
- 破甲
- 充能
- 眩晕

这样既能形成组合，又不会过于复杂。

### 8.5 角色职业
MVP 推荐 4 职业：
- **守护者**：坦克 / 护盾 / 嘲讽类
- **战士**：单体输出 / 破甲
- **术士**：群体伤害 / 中毒 / 控场
- **支援者**：治疗 / 增益 / 充能辅助

### 8.6 技能设计原则
每个角色 2 个主动技能 + 1 个被动标签即可。

示例技能模板：
- 单体攻击 + 附加中毒
- 攻击有破甲效果
- 为全队施加护盾
- 回复单体生命并提供充能
- 攻击眩晕低概率触发

### 8.7 敌人设计原则
MVP 敌人不做 AI 驱动，只做程序行为模板：
- 普通敌：单一攻击模式
- 精英敌：两种技能循环
- Boss：三阶段/两阶段简单机制

示例敌人类型：
- 狂化拾荒者
- 裂界兽
- 教团执行者
- 晶核傀儡

### 8.8 战斗与人物的结合
角色性格可影响战斗表现描述，但不直接改底层规则。

例如：
- 冷静角色的技能文案更克制
- 冲动角色的攻击文案更激烈
- 忠诚角色在保护队友时触发特殊语音

## 9. AI 内容生成系统

### 9.1 AI 使用目标
AI 不是实时导演整局，而是按节点生成内容。

### 9.2 AI 介入时机
MVP 仅在以下场景调用 AI：
1. 开局生成本局背景摘要
2. 招募角色时生成角色档案
3. 特定剧情节点生成事件文本
4. 结局时生成冒险编年史

### 9.3 AI 输出要求
所有 AI 输出必须优先产出结构化 JSON，再渲染成文本。

### 9.4 角色生成示例结构
```json
{
  "name": "阿岚",
  "title": "无光街的缝补师",
  "archetype": "mechanic_support",
  "personalityTags": ["冷静", "务实"],
  "background": "曾在碎城下层修补失控装置，后来被教团追捕。",
  "speechStyle": "说话简短，带一点冷幽默。",
  "stanceToPlayer": "中立偏信任"
}
```

### 9.5 事件生成示例结构
```json
{
  "title": "破碎灯塔",
  "summary": "队伍在旧时代灯塔废墟中发现一段仍在运转的信号。",
  "choices": [
    {
      "text": "强行破解装置",
      "effect": ["gain_resource", "risk_battle"]
    },
    {
      "text": "让同伴判断是否安全",
      "effect": ["relationship_check", "safe_reward"]
    }
  ]
}
```

### 9.6 AI 生成约束策略
必须有以下限制：
- 使用固定世界观前提
- 使用固定可选标签池
- 角色职业只能从程序定义中选择
- 技能机制不能由 AI 自由创造
- 每次只生成当前节点所需内容
- 超出规则的内容直接丢弃或修复

### 9.7 Fallback 机制
如果 AI 请求失败：
- 使用本地模板随机生成人物
- 使用预设事件库替代
- 继续游戏，不中断流程

这个机制必须从 MVP 第一版就存在。

## 10. 剧情与事件设计

### 10.1 剧情结构
MVP 不做长主线，只做“本局故事骨架”。

结构：
- 开场引子
- 中途冲突升级
- 关键选择
- Boss 对决
- 结局总结

### 10.2 事件节点类型
MVP 推荐事件分为：
- **角色遭遇**：招募/对话
- **世界事件**：遗迹/教团/商会/灾变
- **道德抉择**：救助、交易、背弃、牺牲
- **风险事件**：资源换取、战斗概率、关系变化

### 10.3 事件设计原则
每个事件必须至少带来一项可验证的结果：
- 资源变化
- 队伍变化
- 关系变化
- 战斗触发
- 状态加成

不能只是看一段文字。

### 10.4 事件与角色反应
同一个事件发生后，当前队伍中的角色可以追加一句态度反馈。

示例：
- 务实角色：“至少我们拿到了有用的东西。”
- 理想主义角色：“你有没有想过，那个孩子其实在等我们帮他？”
- 贪婪角色：“早知道应该再多拿一点。”

这类反馈是“人物驱动感”的关键来源。

## 11. 页面与 UI 结构

### 11.1 页面列表
MVP 建议包含以下页面/界面：

1. **标题页**
2. **开局页**（选择主角原型 / 开始新局）
3. **地图节点页**
4. **事件页**
5. **角色招募页**
6. **战斗页**
7. **营地/商店页**
8. **结局结算页**
9. **图鉴/记录页**（可选）

### 11.2 地图节点页信息
展示内容：
- 当前章节/区域名
- 节点路径
- 队伍成员头像/姓名/血量
- 当前资源
- 下一个可选节点

### 11.3 战斗页信息
展示内容：
- 左侧队伍状态
- 右侧敌人状态
- 当前行动角色
- 技能按钮
- 战斗日志
- 简短角色语音/气泡文本

### 11.4 结算页信息
展示内容：
- 本局标题
- 冒险摘要
- 队伍成员最终评价
- 击败的 Boss
- 做过的关键选择
- 下次解锁内容（若做 meta progression）

### 11.5 UI 风格建议
MVP 建议：
- 以 UI 卡牌式呈现人物和事件
- 地图采用节点连线方式
- 战斗采用静态立绘 + 数值飘字 + 轻特效
- 不追求复杂动画，优先保证信息清晰与开发效率

## 12. 数据结构设计（MVP 级别）

### 12.1 Character
```ts
export type Character = {
  id: string

  background: string
  relationshipToPlayer: 'support' | 'neutral' | 'dislike'
  stats: {
    hp: number
    maxHp: number
    atk: number
    def: number
    spd: number
    sp: number
    maxSp: number
  }
  skills: string[]
```

### 12.2 Skill
```ts

  id: string
  name: string
  cost: number

  chance?: number

```

```ts

  type: 'battle' | 'event' | 'recruit' | 'camp' | 'shop' | 'boss'

  id: string
```

## 13. 前端技术架构建议

### 13.1 推荐技术栈
- **框架**：React + TypeScript
- **状态管理**：Zustand
- **游戏演出/UI**：React 为主
- **若需要轻战斗演出**：可引入 Phaser，但 MVP 阶段也可先不用
- **样式**：Tailwind CSS 或 CSS Modules
- **本地存档**：localStorage（MVP）
- **数据校验**：Zod

### 13.2 为什么这样选
- React 上手快，适合以界面和状态为核心的游戏
- 回合制和节点推进并不一定需要复杂游戏引擎
- 纯前端更容易快速迭代
- Zustand 足够轻，适合管理战斗、队伍、地图状态

### 13.3 模块划分建议
```txt
src/
  core/
    battle/
    event/
    progression/
    save/
  content/
    static/
    templates/
    schemas/
    generators/
  state/
    gameStore.ts
    battleStore.ts
    uiStore.ts
  services/
    aiClient.ts
    aiCache.ts
    validators.ts
  ui/
    screens/
    components/
```

## 14. 存档与容错设计

### 14.1 存档范围
MVP 存档内容：
- 当前局进度
- 当前队伍
- 当前资源
- 当前节点地图
- 已触发的剧情事实
- AI 已生成内容缓存

### 14.2 存档格式
```ts
export type SaveFile = {
  version: number
  savedAt: number
  runState: RunState
}
```

### 14.3 容错要求
必须处理：
- AI 返回字段缺失
- AI 返回非法值
- 本地存档损坏
- 旧版本存档字段不兼容

### 14.4 容错策略
- 所有 AI 输出先过 schema
- 不合法则修复或 fallback
- 每次节点结束自动存档
- Boss 前额外存档一次

## 15. 内容产量建议（MVP）

### 15.1 最小内容需求
建议最少准备：
- 主角原型：3 个
- 角色原型模板：8 个
- 敌人模板：10 个
- 技能模板：16~20 个
- 事件模板：12 个
- Boss：2 个
- 开局背景模板：3 个
- 结局模板：4 个

### 15.2 为什么不是全 AI
因为全 AI 会导致：
- 不稳定
- 不可控
- 不好调试
- 调性不统一

所以 MVP 应采用：
- **固定骨架 + AI 变体**

## 16. 开发优先级

### Phase 1：无 AI 骨架
完成：
- 地图节点流程
- 战斗系统
- 角色与敌人静态模板
- 存档
- 一局闭环

### Phase 2：接入 AI 内容生成
完成：
- 角色生成
- 事件文案生成
- 结局编年史生成
- JSON schema 校验
- fallback

### Phase 3：人物驱动增强
完成：
- 角色态度反馈
- 关系变化
- 特殊战斗语音
- 结局角色评价

---

## 17. 风险与规避策略

### 风险 1：AI 生成内容不稳定
规避：
- schema + fallback
- 标签池约束
- 世界规则固定

### 风险 2：战斗不好玩
规避：
- 先用静态角色把战斗打磨到可玩
- AI 最后接入文案层

### 风险 3：项目膨胀
规避：
- 坚持单局制
- 不做开放世界
- 不做复杂长线养成

### 风险 4：开发周期过长
规避：
- 用 UI + 数据驱动代替复杂演出
- 优先可玩，不优先炫技

---

## 18. MVP 验收标准

满足以下条目即可进入下一阶段：
- 可以开始新局
- 可以推进完整节点流程
- 可以招募至少 2 名 AI 生成角色
- 可以进行至少 5 场战斗
- 可以打完 Boss 并看到结局总结
- 队伍角色对至少 3 类事件有反馈
- AI 故障时可自动回退模板内容

---

## 19. 后续扩展方向
MVP 完成后，可考虑扩展：
- 多世界主题
- 多结局结构
- 角色图鉴
- 长线解锁系统
- 更多职业与状态机制
- 角色羁绊连携
- PWA / 移动端优化
- 本地模型辅助生成

---

## 20. 最终建议

如果这个项目要真的做出来，正确顺序一定是：

1. **先把它做成一个好玩的回合制节点冒险游戏**
2. **再让 AI 让人物“活起来”**
3. **最后再追求更自由的内容生成**

不要一开始追求“无限剧情”。
先做出“这一局的 3 个角色让我记住了”，这个项目就有价值。

---

## 21. 一句话版本结论

**《无限世界》MVP 应该被定义为：一个纯前端、单局制、节点推进、3 人小队回合制、以 AI 生成人物与事件反馈为特色的轻 roguelite RPG。**

它的核心卖点不是“什么都能生成”，而是：

**每一局，你都会和一组不同的人，一起经历一段还能打、还能记住的冒险。**
