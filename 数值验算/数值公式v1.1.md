### 2.2 六维职责口径

以下职责来自 [`无限世界_数值与战斗系统初稿_v3.md`](数值验算/无限世界_数值与战斗系统初稿_v3.md:111)，且已在测试平台中落实到导出公式：

- `STR`：偏向生命、物攻、物防
- `AGI`：偏向速度、命中、闪避、少量物攻与暴击

- `SPI`：偏向生命、SP、法防、异常攻防、治疗与续航
- `CHA`：偏向统御、治疗辅助、`LEAD`
- `LUK`：偏向暴击、暴伤、闪避、少量异常攻防与速度

### 2.4 数值计算通用约定

```text

round(x, digits) = 仅用于测试展示/报表，不属于战斗核心规则
```

## 4. 已验证命中 / 伤害 / 暴击 / 控制 / 治疗 / 护盾公式

### 4.1 命中公式

```text
HitChance = clamp(90 + HIT - EVA, 72, 98)
```

已验证采用：
- 命中存在固定下限 `72%`
- 命中存在固定上限 `98%`

### 4.2 伤害公式

```text
Mitigated = max(0, ATK * SkillRate - DEF)

FinalDamageRaw = Mitigated * RandomFactor * FinalBonus * CritMultiplier * StageDamageMultiplier * StageTakenMultiplier

FinalDamage = floorMin(FinalDamageRaw, 1)
```

字段映射：

```text
ATK                = PATK 或 MATK，取决于技能 scaling
DEF                = PDEF 或 MDEF，取决于技能 damageType
SkillRate          = 技能倍率
RandomFactor       = 随机波动；公式测试固定为 1，战斗模拟为 0.92 ~ 1.08
FinalBonus         = 团队光环等最终乘区，当前测试平台主要由 TeamAuraBonus 提供
CritMultiplier     = 暴击倍率，非暴击为 1
StageDamageMultiplier / StageTakenMultiplier = 阶段压制乘区
```

已验证采用：
- 若 `ATK * SkillRate <= DEF`，仍保底造成 `1` 点伤害
- 最终伤害先完成全部乘算，再向下取整

当前优化建议：
- 现行模型为**纯减法防御**，实现简单、样例稳定，但会出现“高防废攻 / 低防无感”两端问题
- 后续若要优化，可考虑改为“减法 + 比例”混合防御，避免大量攻击被直接打成保底 `1`
- 由于这会影响所有历史样例与 build 强弱排序，属于**需整体联调的大改项**，不建议在未补足回归测试前直接替换

### 4.3 暴击公式

暴击判定与倍率在测试平台中由 [`computeCritMultiplier()`](数值验算/battle_test_platform.html:686) 拆为两步：

```text
CritChance = clamp(CRIT + SkillExtraCrit, 0, 95)

CritDamageMultiplier = (CDMG + SkillExtraCritDamage) / 100

IfCrit = random(0,100) < CritChance

CritMultiplier = IfCrit ? CritDamageMultiplier : 1
```

已验证采用：
- 暴击率上限为 `95%`
- `CDMG` 的语义是百分比，例如 `150` 表示 `1.5x`
- 技能可额外附加暴击率与暴伤修正
- `CritBonus` 与 `CritDamageBonus` 已作为平面输入字段存在，可由装备、词条、被动、Buff 等来源提供
- 技能侧也可通过 `SkillExtraCrit` / `SkillExtraCritDamage` 做额外修正

补充说明：
- 基础六维导出的 `CRIT` 并不是暴击体系的全部来源，装备、技能、被动与临时状态都会显著影响实际战斗暴击表现
- 例如 [`critGamble`](数值验算/battle_test_platform.html:477) 与 [`burstFinisher`](数值验算/battle_test_platform.html:478) 都直接附带额外暴击率与暴击伤害修正
- 因此不能只根据裸属性下的 `CRIT` 样例，直接断言暴击流后期一定偏弱；应把装备与技能额外项一起纳入完整 build 评估

当前优化建议：
- 建议在后续文档或测试中补充“裸面板暴击”和“实战有效暴击”两套观察口径
- 建议补充高暴击 build 的整套样例：裸 `CRIT`、装备修正后 `CRIT`、技能修正后 `CritChance`、暴击后的 `CritDamageMultiplier`

### 4.4 控制命中公式

```text
StatusChanceRaw = BaseChance * sqrt(max(0.0001, AACC) / max(0.0001, ARES)) * StageControlOutMultiplier * StageControlInMultiplier

StatusChance = clamp(StatusChanceRaw, 5, 85)
```

已验证采用：
- 控制成功率最低为 `5%`
- 控制成功率最高为 `85%`
- 阶段压制当前同时作用于 `control_out` 与 `control_in` 两个乘区

当前实现复查结论：
- [`statusChance()`](数值验算/battle_test_platform.html:283) 的写法是把 `control_out` 与 `control_in` **一起乘在最终概率上**
- [`getStageMultiplier()`](数值验算/battle_test_platform.html:221) 在 `realmGap < 0` 时会交换 `control_out` 与 `control_in`
- 但由于两者最终都是相乘，交换顺序不会改变乘积结果

因此当前平台中的控制阶段压制，实际效果是：

```text
R1: 0.85 * 0.85 = 0.7225
R2: 0.45 * 0.35 = 0.1575
R3: 0.10 * 0.10 = 0.01
```

这意味着：
- 当前实现里，控制压制是**按阶段差绝对值生效**
- 对于控制成功率，`realmGap=+2` 与 `realmGap=-2` 的乘积完全一致
- 它并没有形成“高阶施控更容易 / 低阶施控更困难”的方向性差异，而是形成了“只要存在阶段差，控制整体都更难成立”的效果

这不是公式抄写错误，而是当前公式结构本身的结果。

优化建议：
- 若目标是“双方各有加成折扣”，则应拆成**施加方乘区**与**承受方乘区**的显式双边模型，而不是把两个乘区预先合并后统一相乘
- 推荐表达为：

```text
StatusChanceRaw = BaseChance
                * sqrt(max(0.0001, AACC) / max(0.0001, ARES))
                * AttackerControlOutBonus
                * TargetControlInBonus
```

- 其中：
  - `AttackerControlOutBonus` 表示施加方位格对“施控能力”的加成/折扣
  - `TargetControlInBonus` 表示目标位格对“被控倾向”的加成/折扣
- 这样文档与代码都能明确体现“谁获得加成、谁承受折扣”

### 4.5 治疗公式

```text
FinalHeal = max(0, floor(HEALP * SkillRate * FinalBonus * StageHealShieldMultiplier))
```

已验证采用：
- 治疗值向下取整
- 治疗结果最小为 `0`
- 实际入血量还需受目标缺失生命限制

战斗中的有效治疗量：

```text
ActualHeal = min(TargetMissingHP, FinalHeal)
```

### 4.6 护盾公式

```text
FinalShield = max(0, floor((HEALP * SkillRate + FlatShield) * FinalBonus * StageHealShieldMultiplier))
```

已验证采用：
- 护盾与治疗共享 `heal_shield` 阶段乘区
- 护盾值向下取整
- 护盾结果最小为 `0`

### 4.7 护盾结算规则

已验证采用：

```text
1. 同 sourceId 的护盾重复施加时，旧护盾被新护盾覆盖
2. 不同 sourceId 的护盾最多同时保留 3 层
3. 护盾列表按 value 从大到小排序，超出 3 层时移除较小层
4. 护盾按 remainingSelfActions 计时，持续 3 次“自身行动后”
5. 受到伤害时按当前护盾列表顺序逐层吸收
6. value <= 0 或 remainingSelfActions <= 0 的护盾会被移除
```

## 5. 已验证 ATB / SP / Burst / 阶段压制 / Boss 人数补偿规则

### 5.1 ATB 行动条规则

```text
InitialGauge = 0

EffectiveSPD = RawSPD                              // 默认
EffectiveSPD = min(220, RawSPD) + max(0, RawSPD - 220) * 0.5   // 启用软上限时

PerTickGaugeGain = EffectiveSPD * GaugeBonus

当 Gauge >= 1000 时：
  Gauge -= 1000
  获得 1 次行动
```

已验证采用：
- 测试平台 ATB 初始条固定为 `0`
- 行动阈值固定为 `1000`
- 高速单位在启用软上限时，不应对普通单位形成过高行动比

当前优化建议：
- `220` 的软上限阈值已能抑制极端高敏 build，但其设计锚点尚未在文档中解释清楚
- 若未来实战中发现高敏 build 被压得过重，可优先调整阈值或折算斜率，而不是改动整个 ATB 框架

### 5.2 SP 资源循环规则

```text
行动前判断是否够支付 SkillCost
若足够：先扣除 SkillCost
无论是否成功施放技能，行动结束后：
SP = min(MaxSP, SP + SPRegenPerAction)
```

已验证采用：
- SP 回复发生在“每次自身行动后”
- 即使 `SP=0`，角色也不会死锁；仍可完成行动并在行动后回复

### 5.3 Burst 资源规则

测试平台使用固定事件映射：

```text
basicHit   => +12
skillUse   => +8
takeDamage => +6
kill       => +20
crit       => +6
```

基础模拟规则：

```text
Burst = min(100, Burst + Gain)
Spend 时：Burst = max(0, Burst - Cost)
```

```text
  control_in: 1.00,
  heal_shield: 1.00
}

R1 = {
  damage: 1.25,
  taken: 0.80,
  control_out: 0.85,
  control_in: 0.85,
  heal_shield: 1.10
}

R2 = {
  damage: 1.70,
  taken: 0.45,
  control_out: 0.45,
  control_in: 0.35,
  heal_shield: 1.18
}

R3 = {
  damage: 2.40,
  taken: 0.20,
  control_out: 0.10,
  control_in: 0.10,
  heal_shield: 1.25
}
```

当前平台换算规则由 [`getStageMultiplier()`](数值验算/battle_test_platform.html:221) 实现：

```text
当 realmGap = 0：直接使用对应档位
当 realmGap > 0：表示施加/攻击方高阶，直接使用对应档位
当 realmGap < 0：
  damage      = preset.taken
  taken       = preset.damage
  control_out = preset.control_in
  control_in  = preset.control_out
  heal_shield = 1 / preset.heal_shield
```

#### 5.4.1 对实际公式结构的复查结论

这里必须区分“参数定义”与“最终参与公式的方式”。

**伤害公式实际是双方各有乘区的：**

```text
FinalDamageRaw = Mitigated
               * RandomFactor
               * FinalBonus
               * CritMultiplier
               * StageDamageMultiplier
               * StageTakenMultiplier
```

因此在伤害线上，阶段压制已经具备“双方各有加成折扣”的结构：
- `damage` 更像攻击方乘区
- `taken` 更像承受方乘区

把它展开后，双方表现如下：

##### 当攻击方高阶（realmGap > 0）

```text
攻击方获得：damage 加成
防守方获得：taken 折扣
```

详细为：

```text
R1: 攻击方 damage=1.25，防守方 taken=0.80
R2: 攻击方 damage=1.70，防守方 taken=0.45
R3: 攻击方 damage=2.40，防守方 taken=0.20
```

##### 当攻击方低阶（realmGap < 0）

```text
攻击方获得：damage 折扣（来自对方档位的 taken）
防守方获得：taken 加成（来自对方档位的 damage）
```

详细为：

```text
R1: 攻击方 damage=0.80，防守方 taken=1.25
R2: 攻击方 damage=0.45，防守方 taken=1.70
R3: 攻击方 damage=0.20，防守方 taken=2.40
```

从“双方各有乘区”的结构来看，伤害线是成立的。

#### 5.4.2 当前伤害线的实际问题不在“有没有双边乘区”，而在“乘积目标值”

把双方乘区相乘后，现平台得到：

```text
realmGap = +1  => 1.25 * 0.80 = 1.00
realmGap = +2  => 1.70 * 0.45 = 0.765
realmGap = +3  => 2.40 * 0.20 = 0.48

realmGap = -1  => 0.80 * 1.25 = 1.00
realmGap = -2  => 0.45 * 1.70 = 0.765
realmGap = -3  => 0.20 * 2.40 = 0.48
```

这说明：
- 当前平台的伤害阶段压制是**双边结构正确，但净乘积按绝对阶段差对称**
- 它并没有形成“高阶打低阶净伤害更高、低阶打高阶净伤害更低”的方向性差异
- 而是形成了“有阶段差时，双方的最终伤害净值一起被压缩到某个固定倍率”的效果

换句话说：
- **结构上**：当前公式已经是双方各有加成/折扣
- **效果上**：当前参数组没有做出预期的方向性结果

这是本轮复查后需要明确修正的结论。

#### 5.4.3 若要保留“双边乘区”结构，推荐的优化方向

若目标是：
- 高阶打低阶：伤害更容易成立
- 低阶打高阶：伤害明显受压制
- 双方仍各自保留独立的加成/折扣字段

则建议把 `damage` 与 `taken` 视为**面向不同一侧的独立参数**，但不要再要求其镜像后乘积对称。

可采用下述目标口径：

```text
高阶打低阶：
- 攻击方 damage > 1
- 防守方 taken >= 1 或小幅 < 1
- 最终净乘积应 > 1

低阶打高阶：
- 攻击方 damage < 1
- 防守方 taken > 1
- 最终净乘积应明显 < 1
```

例如仅作为设计方向示意：

```text
R1 正差：damage=1.15, taken=0.95 => 净 1.0925
R1 负差：damage=0.85, taken=1.10 => 净 0.935

R2 正差：damage=1.35, taken=0.90 => 净 1.215
R2 负差：damage=0.55, taken=1.35 => 净 0.7425

R3 正差：damage=1.60, taken=0.85 => 净 1.36
R3 负差：damage=0.25, taken=1.80 => 净 0.45
```

注意：
- 上述只是说明“公式结构应该如何体现双方各有乘区且方向不同”
- 不是要求立即替换为这一组具体数值
- 若未来要实装，应把样例、Boss、治疗、控制一起重测

#### 5.4.4 控制线若要做到双方各有加成折扣，应如何表达

当前控制线的问题不是缺参数，而是参数被合并后失去了方向性。

建议在规则文档中把控制阶段压制显式写成：

```text
StatusChanceRaw = BaseChance
                * sqrt(max(0.0001, AACC) / max(0.0001, ARES))
                * AttackerControlOutMultiplier
                * TargetControlInMultiplier
```

并明确：

```text
当施加方高阶时：
- AttackerControlOutMultiplier 应倾向 > 1 或接近 1
- TargetControlInMultiplier 应倾向 > 1（更容易被控）或至少不额外压低

当施加方低阶时：
- AttackerControlOutMultiplier 应明显 < 1
- TargetControlInMultiplier 应明显 < 1（高阶更难被控）
```

如果继续沿用当前 `R1/R2/R3` 数字，也应至少将文档解释为：
- 现平台控制阶段压制是“绝对阶段差惩罚”
- 不是“方向性强弱压制”

否则容易造成设计理解偏差。

### 5.5 Boss 人数补偿规则

```text
EnemyCountAdvantage = 敌方人数 - Boss 所在方人数

BossHPBonus = 1 + 0.35 * EnemyCountAdvantage
BossStatusResistBonus = 1 + 0.20 * EnemyCountAdvantage
BossActionGaugeBonus = 1 + 0.15 * EnemyCountAdvantage
```

已验证采用：
- `BossActionGaugeBonus` 直接乘到每 tick 的 `ATB` 获取速度上
- 因此同 `SPD` 下，Boss 会比普通单位更快出手

## 7. 当前公式的优化空间汇总

本章用于补充 [`数值公式v1.md`](数值验算/数值公式v1.md) 中尚未展开的风险点与优化方向。

### 7.1 SPI 覆盖面过广，存在“万金油”风险

当前 `SPI` 同时影响：

```text
MaxHP
MaxSP
MATK
MDEF
SPD
AACC
ARES
HEALP
LEAD
SPRegenPerAction
```

结论：
- `SPI` 是当前系统中覆盖导出面最广的属性
- 无论法系、治疗、控制、统御还是续航构筑，投入 `SPI` 通常都不亏
- 这容易让其成为泛用副核心甚至压过部分主属性

优化建议：
- 后续如要收窄其泛用性，可优先下调其对 `SPD`、`AACC`、`LEAD` 的次级贡献
- 让 `SPI` 更聚焦于 `SP / MDEF / HEALP / ARES` 主线，而不是同时侵入太多系统

### 7.2 `TeamAuraBonus` 与 `SummonInheritance` 封顶过早

当前为：

```text
TeamAuraBonus = 1 + min(0.20, LEAD / 1000)
SummonInheritance = 0.5 + min(0.35, LEAD / 1000)
```

结论：
- `TeamAuraBonus` 在 `LEAD = 200` 即封顶
- `SummonInheritance` 在 `LEAD = 350` 即封顶
- 中期统御向角色就可能接近上限，后续 `CHA/LEAD` 继续投入的边际收益下降过快

优化建议：
- 可提高封顶所需 `LEAD`
- 或改为缓慢递减曲线，避免中期直接撞顶

### 7.3 `EVA` 无基础常数，低敏目标极易被命中

当前为：

```text
EVA = AGI * 0.9 + LUK * 0.6 + Level * 0.1 + EvadeBonus
```

结论：
- 相比 `HIT` 的 `100` 基础值，`EVA` 没有基础常数
- 完全不投敏捷/幸运的目标，实战里会非常接近 `98%` 被命中上限

这本身未必错误，但建议明确为设计意图，否则容易让玩家误解“闪避属性几乎没起点”。

### 7.4 ATB 软上限阈值 220 需要补充设计锚点说明

当前公式已经可用，但文档里缺少对 `220` 的解释。

优化建议：
- 补充这个阈值对应的目标档位、典型角色速度段与预期行动比
- 便于后续平衡调整时知道应该调“阈值”还是调“折算斜率”

### 7.5 暴击体系需要区分“裸属性导出”与“实战构筑表现”

结论：
- 当前 `CRIT`、`CDMG` 的裸属性公式并不代表实战暴击 build 的完整上限
- 装备、词条、被动、技能额外暴击与额外暴伤都已纳入平台字段与技能定义
- 因此后续评估暴击流强弱，必须把 `CritBonus`、`CritDamageBonus`、`SkillExtraCrit`、`SkillExtraCritDamage` 一起看

建议补充样例：

```text
裸属性暴击样例
装备后暴击样例
技能启动时暴击样例
暴击后总倍率样例
```

### 7.6 召唤体系文档可补充完整继承表

测试平台中，召唤物不仅继承 `SummonInheritance`，还对各战斗属性有二次折算，见召唤生成逻辑 [`summonStats`](数值验算/battle_test_platform.html:618)。

建议后续文档补充：
- `PATK / MATK / PDEF / MDEF / SPD / HIT / EVA / CRIT / AACC / ARES / HEALP` 的实际继承比例
- 便于召唤玩法做更稳定的策划和验算

## 9. 当前版本使用结论

本文结论如下：

```text
- 基础导出、命中、伤害、暴击、治疗、护盾、ATB、SP、Burst、Boss 补偿：当前实现可直接沿用
- 纯减法防御：可继续使用，但已确认存在中后期缩放风险
- 阶段压制伤害线：当前已具备双方双乘区结构，但参数结果呈绝对值对称，尚未形成明确方向性压制
- 阶段压制控制线：当前实现为绝对阶段差惩罚，未真正体现“高阶更易控低阶 / 低阶更难控高阶”
- SPI 覆盖面、LEAD 封顶节奏、EVA 起点、ATB 软上限锚点：都有继续优化空间
- 暴击体系评估必须纳入装备、技能、词条等额外来源，不能只看裸属性导出
- 成长方案 A、最小技能循环、R3 固定占位参数：仍属于暂定假设
```

因此，[`数值公式v1.1.md`](数值验算/数值公式v1.1.md) 可作为当前阶段更准确的复查版规范：
- 对“已验证采用”的部分继续沿用
- 对“存在结构性优化空间”的部分明确标注风险与推荐方向
- 在正式改动阶段压制与控制压制前，应先补足正负阶段差的自动化回归测试
