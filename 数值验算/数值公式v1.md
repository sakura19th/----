# 数值公式 v1

## 1. 文档定位与适用范围

本文档用于把测试平台中**已经验证并采用**的数值公式、战斗规则与工程化处理方式，整理为后续游戏实现可直接遵循的统一规范。

适用对象：
- 程序实现战斗结算、角色导出、资源系统、Boss 规则时的落地依据
- 策划编写技能、怪物、数值表与测试用例时的共同参照
- 后续自动化测试、对战回放校验、验算脚本的基准说明

事实来源仅限以下 3 份现有材料：
- [`battle_test_platform.html`](battle_test_platform.html)
- [`无限世界_数值与战斗系统初稿_v3.md`](无限世界_数值与战斗系统初稿_v3.md)
- [`无限世界_数值与战斗系统_v3测试方案.md`](无限世界_数值与战斗系统_v3测试方案.md)

本文**不引入新的未验证设计**。凡是测试平台明确采用且有测试样例支撑的内容，标记为“已验证采用”；凡是测试平台为保证原型可运行而显式写入、但目前更偏工程占位或样机假设的内容，标记为“暂定假设”。

## 2. 术语与基础输入定义

### 2.1 基础输入字段

角色基础输入以六维属性、等级与若干平面修正项构成，测试平台实际使用字段见 [`battle_test_platform.html`](battle_test_platform.html)。

```text
Level
STR / AGI / INT / SPI / CHA / LUK
FlatHPBonus / FlatSPBonus
WeaponATK / FlatPATKBonus
WeaponMATK / FlatMATKBonus
ArmorBonus / ResistBonus / SpeedBonus
HitBonus / EvadeBonus
CritBonus / CritDamageBonus
StatusHitBonus / StatusResistBonus
HealBonus / LeadBonus
RegenBonus
```

### 2.2 六维职责口径

以下职责来自 [`无限世界_数值与战斗系统初稿_v3.md`](无限世界_数值与战斗系统初稿_v3.md)，且已在测试平台中落实到导出公式：

- `STR`：偏向生命、物攻、物防
- `AGI`：偏向速度、命中、闪避、少量物攻与暴击
- `INT`：偏向法攻、命中、异常命中、少量暴伤、护盾/机制技能收益
- `SPI`：偏向生命、SP、法防、异常攻防、治疗与续航
- `CHA`：偏向统御、治疗辅助、`LEAD`
- `LUK`：偏向暴击、暴伤、闪避、少量异常攻防与速度

### 2.3 派生术语

```text
MaxHP                最大生命
MaxSP                最大技能资源
PATK / MATK          物理攻击 / 法术攻击
PDEF / MDEF          物理防御 / 法术防御
SPD                  速度
HIT / EVA            命中 / 闪避
CRIT / CDMG          暴击率 / 暴击伤害倍率百分比
AACC / ARES          异常命中 / 异常抗性
HEALP                治疗强度
LEAD                 统御值
TeamAuraBonus        团队光环加成
SummonInheritance    召唤继承系数
SPRegenPerAction     每次自身行动后的 SP 回复
realmGap             双方阶段差，正值表示攻击/施加者高阶
```

### 2.4 数值计算通用约定

```text
clamp(x, min, max) = 把 x 限制在 [min, max] 区间
floorMin(x, minValue) = 向下取整后，再保证不低于 minValue
round(x, digits) = 仅用于测试展示/报表，不属于战斗核心规则
```

## 3. 已验证基础导出公式

以下公式已在测试平台中实现，并有明确公式测试样例验证，可直接作为程序导出规范。

### 3.1 基础导出公式

```text
MaxHP = 100 + STR * 9 + SPI * 5 + Level * 4 + FlatHPBonus

MaxSP = 40 + INT * 4 + SPI * 8 + Level * 2 + FlatSPBonus

PATK = 10 + STR * 2.6 + AGI * 0.7 + Level * 0.5 + WeaponATK + FlatPATKBonus

MATK = 10 + INT * 2.6 + SPI * 0.8 + Level * 0.5 + WeaponMATK + FlatMATKBonus

PDEF = 5 + STR * 1.2 + AGI * 0.5 + Level * 0.3 + ArmorBonus

MDEF = 5 + SPI * 1.6 + INT * 0.6 + Level * 0.3 + ResistBonus

SPD = 80 + AGI * 2.2 + SPI * 0.3 + LUK * 0.2 + SpeedBonus

HIT = 100 + AGI * 1.0 + INT * 0.4 + Level * 0.2 + HitBonus

EVA = AGI * 0.9 + LUK * 0.6 + Level * 0.1 + EvadeBonus

CRIT = 5 + AGI * 0.08 + LUK * 0.18 + CritBonus

CDMG = 150 + STR * 0.12 + INT * 0.12 + LUK * 0.10 + CritDamageBonus

AACC = 100 + INT * 0.8 + SPI * 0.8 + CHA * 0.6 + LUK * 0.2 + StatusHitBonus

ARES = 100 + SPI * 1.2 + CHA * 0.5 + LUK * 0.3 + StatusResistBonus

HEALP = SPI * 1.8 + INT * 0.7 + CHA * 0.7 + HealBonus

LEAD = CHA * 1.8 + SPI * 0.6 + LUK * 0.2 + LeadBonus
```

### 3.2 由 LEAD 导出的战斗用附加值

```text
TeamAuraBonus = 1 + min(0.20, LEAD / 1000)

SummonInheritance = 0.5 + min(0.35, LEAD / 1000)

SPRegenPerAction = 4 + SPI * 0.15 + RegenBonus
```

说明：
- `TeamAuraBonus` 与 [`无限世界_数值与战斗系统初稿_v3.md`](无限世界_数值与战斗系统初稿_v3.md) 中“让 LEAD 进入战斗公式”的方向一致，并已在测试平台正式接入。
- `SummonInheritance` 为测试平台固定工程化规则，已被公式样例验证。
- `SPRegenPerAction` 已被资源循环模拟使用。

### 3.3 已验证样例结论

测试平台已直接验证以下代表样例：

## 4. 已验证命中 / 伤害 / 暴击 / 控制 / 治疗 / 护盾公式

### 4.1 命中公式

```text
HitChance = clamp(90 + HIT - EVA, 72, 98)
```

已验证采用：
- 命中存在固定下限 `72%`
- 命中存在固定上限 `98%`

测试平台样例：

```text
HIT-LOW : hit=100, eva=300 => 72
HIT-HIGH: hit=260, eva=10  => 98
```

### 4.2 伤害公式

```text
Mitigated = max(0, ATK * SkillRate - DEF)

FinalDamageRaw = Mitigated * RandomFactor * FinalBonus * CritMultiplier * StageDamageMultiplier * StageTakenMultiplier

FinalDamage = floorMin(FinalDamageRaw, 1)
```

字段映射：

```text
ATK                = PATK 或 MATK，取决于技能 scaling
DEF                = PDEF 或 MDEF，取决于技能 damageType
SkillRate          = 技能倍率
RandomFactor       = 随机波动；公式测试固定为 1，战斗模拟为 0.92 ~ 1.08
FinalBonus         = 团队光环等最终乘区，当前测试平台主要由 TeamAuraBonus 提供
CritMultiplier     = 暴击倍率，非暴击为 1
StageDamageMultiplier / StageTakenMultiplier = 阶段压制乘区
```

已验证采用：
- 若 `ATK * SkillRate <= DEF`，仍保底造成 `1` 点伤害
- 最终伤害先完成全部乘算，再向下取整

### 4.3 暴击公式

暴击判定与倍率在测试平台中拆为两步：

```text
CritChance = clamp(CRIT + SkillExtraCrit, 0, 95)

CritDamageMultiplier = (CDMG + SkillExtraCritDamage) / 100

IfCrit = random(0,100) < CritChance

CritMultiplier = IfCrit ? CritDamageMultiplier : 1
```

已验证采用：
- 暴击率上限为 `95%`
- `CDMG` 的语义是百分比，例如 `150` 表示 `1.5x`
- 技能可额外附加暴击率与暴伤修正

### 4.4 控制命中公式

```text
StatusChanceRaw = BaseChance * sqrt(max(0.0001, AACC) / max(0.0001, ARES)) * StageControlOutMultiplier * StageControlInMultiplier

StatusChance = clamp(StatusChanceRaw, 5, 85)
```

已验证采用：
- 控制成功率最低为 `5%`
- 控制成功率最高为 `85%`
- 阶段压制同时作用于 `control_out` 与 `control_in` 两个乘区

测试平台样例：

```text
CTRL-BASIC  : base=40, aacc=220, ares=220 => 40
CTRL-HARD   : base=40, aacc=320, ares=180 => 53.3
CTRL-EXTREME: base=65, aacc=420, ares=140 => 85
REALM-CTRL-R3: base=40, aacc=320, ares=180, realmGap=3 => 5
```

### 4.5 治疗公式

```text
FinalHeal = max(0, floor(HEALP * SkillRate * FinalBonus * StageHealShieldMultiplier))
```

已验证采用：
- 治疗值向下取整
- 治疗结果最小为 `0`
- 实际入血量还需受目标缺失生命限制

战斗中的有效治疗量：

```text
ActualHeal = min(TargetMissingHP, FinalHeal)
```

测试平台样例：

```text
HEAL-BASIC: healPower=250, skillRate=1.2 => 300
HEAL-MIN-01: healPower=-50, skillRate=1 => 0
```

### 4.6 护盾公式

```text
FinalShield = max(0, floor((HEALP * SkillRate + FlatShield) * FinalBonus * StageHealShieldMultiplier))
```

已验证采用：
- 护盾与治疗共享 `heal_shield` 阶段乘区
- 护盾值向下取整
- 护盾结果最小为 `0`

测试平台样例：

```text
SHIELD-BASIC : healPower=220, skillRate=1.0, flatShield=80 => 300
SHIELD-MIN-01: healPower=-100, skillRate=1, flatShield=-20 => 0
```

### 4.7 护盾结算规则

已验证采用：

```text
1. 同 sourceId 的护盾重复施加时，旧护盾被新护盾覆盖
2. 不同 sourceId 的护盾最多同时保留 3 层
3. 护盾列表按 value 从大到小排序，超出 3 层时移除较小层
4. 护盾按 remainingSelfActions 计时，持续 3 次“自身行动后”
5. 受到伤害时按当前护盾列表顺序逐层吸收
6. value <= 0 或 remainingSelfActions <= 0 的护盾会被移除
```

对应测试验证：

```text
SHIELD-STACK-01: 同源覆盖
SHIELD-STACK-02: 异源最多 3 层
SHIELD-DURATION : 持续 3 次自身行动
```

## 5. 已验证 ATB / SP / Burst / 阶段压制 / Boss 人数补偿规则

## 5.1 ATB 行动条规则

```text
InitialGauge = 0

EffectiveSPD = RawSPD                              // 默认
EffectiveSPD = min(220, RawSPD) + max(0, RawSPD - 220) * 0.5   // 启用软上限时

PerTickGaugeGain = EffectiveSPD * GaugeBonus

当 Gauge >= 1000 时：
  Gauge -= 1000
  获得 1 次行动
```

已验证采用：
- 测试平台 ATB 初始条固定为 `0`
- 行动阈值固定为 `1000`
- 高速单位在启用软上限时，不应对普通单位形成过高行动比

测试平台样例：

```text
100 tick 下：
SPD80  => 8 次行动
SPD120 => 12 次行动
SPD160 => 16 次行动
SPD220 => 22 次行动
SPD300 => 30 次行动

软上限折算：
220 => 220
260 => 240
320 => 270
400 => 310

高敏限制：260 SPD vs 120 SPD，在 softCap=true 时行动次数比 <= 2.2
```

### 5.2 SP 资源循环规则

```text
行动前判断是否够支付 SkillCost
若足够：先扣除 SkillCost
无论是否成功施放技能，行动结束后：
SP = min(MaxSP, SP + SPRegenPerAction)
```

已验证采用：
- SP 回复发生在“每次自身行动后”
- 即使 `SP=0`，角色也不会死锁；仍可完成行动并在行动后回复

测试平台样例：

```text
SP-CYCLE-01: initialSP=100, regen=10, 连续 5 次 cost=8 => finalSP=100
SP-CYCLE-02: initialSP=90, regen=8, 连续 6 次 cost=24 => 成功施放次数落在 4~6 次
SP-ZERO    : initialSP=0, regen=4, cost=10 => usedSkill=false，但流程不死锁
```

### 5.3 Burst 资源规则

测试平台使用固定事件映射：

```text
basicHit   => +12
skillUse   => +8
takeDamage => +6
kill       => +20
crit       => +6
```

基础模拟规则：

```text
Burst = min(100, Burst + Gain)
Spend 时：Burst = max(0, Burst - Cost)
```

战斗中已验证采用：

```text
自身行动后默认获得 +12 Burst
若本次造成暴击，额外 +6 Burst
若承受伤害，可额外获得 +6 Burst
需要消耗 Burst 的技能，独立检查 burstCost，且独立扣除
```

测试平台样例：

```text
BURST-01     : 5 次普攻 => 60
BURST-02     : 5 次技能 => 40
BURST-03     : 6 次受击 => 36
BURST-04     : 2 次击杀 => 40
BURST-EDGE-01: 超上限截断到 100
BURST-EDGE-02: 100 后花费 80 => 剩余 20
```

### 5.4 阶段压制规则

测试平台采用固定阶段预设：

```text
R0 = {
  damage: 1.00,
  taken: 1.00,
  control_out: 1.00,
  control_in: 1.00,
  heal_shield: 1.00
}

R1 = {
  damage: 1.25,
  taken: 0.80,
  control_out: 0.85,
  control_in: 0.85,
  heal_shield: 1.10
}

R2 = {
  damage: 1.70,
  taken: 0.45,
  control_out: 0.45,
  control_in: 0.35,
  heal_shield: 1.18
}

R3 = {
  damage: 2.40,
  taken: 0.20,
  control_out: 0.10,
  control_in: 0.10,
  heal_shield: 1.25
}
```

换算规则：

```text
当 realmGap = 0：直接使用对应档位
当 realmGap > 0：表示施加方高阶，直接使用对应档位
当 realmGap < 0：对 damage/taken/control_out/control_in 做镜像交换，heal_shield 取倒数
```

已验证采用：
- `R1` 下最终伤害乘积约为 `1.25 * 0.8 = 1.0`
- `R2` 下最终伤害乘积约为 `1.7 * 0.45 = 0.765`
- `R3` 下最终伤害乘积固定占位为 `2.4 * 0.2 = 0.48`
- `R3` 控制压制极强，但仍受 `5%` 最低命中保护

测试平台样例：

```text
REALM-01: atk=200, def=100, gap=0 => 100
REALM-02: atk=200, def=100, gap=1 => 100
REALM-03: atk=200, def=100, gap=2 => 76
REALM-04: atk=200, def=100, gap=3 => 48
REALM-CTRL-R3: 高控打高抗在 R3 下仍被夹到 5%
```

### 5.5 Boss 人数补偿规则

测试平台对“单 Boss 对多人”的人数差采用固定补偿：

```text
EnemyCountAdvantage = 敌方人数 - Boss 所在方人数

BossHPBonus = 1 + 0.35 * EnemyCountAdvantage
BossStatusResistBonus = 1 + 0.20 * EnemyCountAdvantage
BossActionGaugeBonus = 1 + 0.15 * EnemyCountAdvantage
```

1v3 对应样例：

```text
enemyCountAdvantage = 2
hpBonus = 1.7
statusResistBonus = 1.4
actionGaugeBonus = 1.3
```

已验证采用：
- `BossActionGaugeBonus` 直接乘到每 tick 的 `ATB` 获取速度上
- 因此同 `SPD` 下，Boss 会比普通单位更快出手

对应测试验证：

```text
BOSS-BONUS-01: 1v3 补偿值正确
BOSS-ATB-01  : gaugeBonus 乘到每 tick 的 ATB 获取速度
```

## 6. 测试平台采用的工程化假设与边界规则

本章分为“已验证采用”和“暂定假设”两类。

### 6.1 已验证采用的工程化规则

以下内容已进入平台逻辑并被测试覆盖，可视为当前实现规范的一部分：

```text
1. 公式测试时 RandomFactor 固定为 1
2. 正式战斗模拟时 RandomFactor = 0.92 ~ 1.08
3. 最终伤害向下取整，且最小为 1
4. 治疗/护盾向下取整，且最小为 0
5. ATB 初始条固定为 0
6. SP 回复发生在每次自身行动后
7. 控制系统当前仅验证“成功率”和“行动压制是否生效”
8. 护盾持续 3 次自身行动
9. 护盾同源覆盖、异源最多 3 层
10. BossActionGaugeBonus 乘在每 tick 的 ATB 获取速度上
11. 极端大数输入（如全属性 999）要求结果仍为有限数值
12. 负值输入不要求平衡，但要求公式可计算、不崩溃
```

### 6.2 暂定假设：已写入平台，但更偏样机占位

以下项目出现在测试平台说明中，当前可作为原型阶段默认值使用，但不应误判为已永久锁定：

#### 6.2.1 成长方案 A

```text
等级档位: 1 / 10 / 30 / 60 / 100
成长系数:
Lv1   => 0.55
Lv10  => 0.78
Lv30  => 1.00
Lv60  => 1.45
Lv100 => 1.95

档位之间采用线性插值
```

说明：
- 该规则已被测试平台用于构筑成长档位导出与对战探针。
- 但它在平台页签中被明确描述为“新增最小假设”，因此本文将其归为“暂定假设”，不是最终成长体系定版。

#### 6.2.2 Build 的最小技能循环

```text
测试平台用最小技能循环来体现 build 定位
不追求完整职业系统与完整技能树
```

说明：
- 此规则适用于测试平台验证“不同 build 能否体现差异”。
- 它是测试与样机组织方式，不等于正式内容量设计规范。

#### 6.2.3 R3 参数的“固定占位”属性

```text
R3 = damage 2.40 / taken 0.20 / control_out 0.10 / control_in 0.10 / heal_shield 1.25
```

说明：
- 虽然当前测试平台已采用并据此验证，但页面说明明确写为“R3 固定占位”。
- 因此当前应理解为“已被当前平台采用的暂定高阶压制参数”。
- 在正式大版本锁定前，不应扩写为更高阶体系或外推更多阶段档。

## 7. 当前已验证结论与未完全验证项

### 7.1 当前已验证结论

#### 7.1.1 基础导出与子公式可稳定计算

```text
MaxHP / MaxSP / PATK / MATK / CRIT / LEAD 派生项均有样例验证
命中、控制、治疗、护盾、伤害保底等核心子公式均可直接验算
极端大数与负值输入不会导致平台崩溃
```

#### 7.1.2 LEAD 已真正进入战斗体系

```text
TeamAuraBonus = 1 + min(0.20, LEAD / 1000)
SummonInheritance = 0.5 + min(0.35, LEAD / 1000)
```

这意味着：
- `CHA`/统御向构筑不再只存在纸面设定价值
- 统御流在多人战和召唤体系中已经具备可验证的战斗收益入口

#### 7.1.3 ATB 节奏已具备基础稳定性

```text
SPD 直接驱动行动密度
```

## 8. 后续实现注意事项

### 8.1 程序实现口径

后续战斗代码、服务端验算或前端重放逻辑，应优先保持与本文一致：

```text
1. 导出公式字段名保持一致，避免策划表与代码口径漂移
2. 伤害、治疗、护盾统一使用“先乘算，后取整”的处理顺序
3. 命中、控制、暴击上/下限必须写成硬限制，不依赖策划手动保证
4. SP 回复时机固定在“每次自身行动后”
5. Burst 与 SP 分开扣费与校验，避免混用资源逻辑
6. 阶段压制统一通过 StageMultiplier 层处理，不要散落在各技能脚本内
7. Boss 人数补偿直接作用在初始化属性和每 tick 行动条获取上
8. 护盾的覆盖、排序、层数上限、持续回合必须封装成独立公共逻辑
```

### 8.2 策划配置口径

```text
1. 技能倍率设计应以本文公式为准，不应再单独发明第二套乘区
2. 若设计新控制技能，应先复用 AACC/ARES + BaseChance 框架
3. 若设计新治疗/护盾技能，应优先复用 HEALP 与 heal_shield 阶段乘区
4. 若设计统御/召唤玩法，应先落在 LEAD -> TeamAuraBonus / SummonInheritance 这两条已验证入口上
5. 若设计 Boss，多人补偿应优先使用现有人数差规则，再决定是否叠加专属机制
```

### 8.3 测试与验收口径
