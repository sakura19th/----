# Stage 0 本地直开兼容性修复说明

## 1. 文档目的

本文档用于记录本次“修复构建产物在本地 `file://` 可直接打开，同时保持 GitHub Pages 可用”这一兼容性修复的背景、根因、方案、落地结果与后续维护约束。

本文面向未来维护者，目标不是复述一次性操作，而是沉淀可持续执行的修改边界，确保后续继续调整构建链路、资源组织或页面结构时，不破坏当前的本地直开兼容性。

相关规范与参考：

- 目录规范见 [`plans/README.md`](plans/README.md)
- 既有阶段记录见 [`plans/Stage0_实施记录与后续修改指南.md`](plans/Stage0_实施记录与后续修改指南.md)

## 2. 问题背景

本次修复关注的是一个典型的静态前端分发兼容性问题：

- 构建产物部署到 GitHub Pages 时需要可正常访问
- 同一份构建产物在本地通过 `file://` 直接打开 [`dist/index.html`](dist/index.html) 时，也需要能够正常加载页面

这个目标比普通的 Vite 默认部署要求更严格。

默认情况下，现代前端构建工具更偏向 HTTP 场景，也就是：

- 通过本地开发服务器访问
- 通过静态站点托管服务访问
- 依赖浏览器在 HTTP/HTTPS 协议下对模块、资源和请求能力的正常支持

但当产物要兼容 `file://` 直接打开时，浏览器行为会变得更保守，尤其是以下能力常出现限制：

- ES Module 脚本加载
- 额外 JS/CSS 文件引用
- 运行时 `fetch`
- 动态导入
- Worker
- WASM
- 外部字体、音频或其他跨路径资源

因此，这次修复的本质不是单点改路径，而是把构建产物从“适合静态托管”进一步调整为“尽量接近单文件、弱依赖协议环境”的输出形式。

---

## 3. 现象与根因分析

### 3.1 修复前的现象

修复前，构建产物虽然能成功生成，但在本地直接打开时容易出现以下问题：

- 页面白屏
- 主脚本未执行
- 样式未生效
- 浏览器控制台中出现模块加载或资源访问失败

### 3.2 根因拆解

根因不是单一问题，而是多个机制叠加：

#### 3.2.1 默认 ESM 输出更依赖浏览器按模块方式加载资源

Vite 默认构建结果倾向于输出 ES Module 形式脚本，并在 [`index.html`](index.html) 中通过模块脚本标签引用产物。该方式在 HTTP/HTTPS 环境下通常没有问题，但在 `file://` 下，浏览器对模块脚本及其后续依赖链的处理更容易触发限制。

#### 3.2.2 默认构建产物通常拆分为多个文件

普通构建会生成：

- 一个 HTML 入口
- 若干 JS 文件
- 若干 CSS 文件
- 可能还有 preload 相关标签

这意味着打开 [`dist/index.html`](dist/index.html) 时，浏览器还需要继续解析并加载多个相对资源。只要其中某个链路在 `file://` 下处理不一致，就会影响页面启动。

#### 3.2.3 绝对路径或面向托管环境的路径策略不适合本地直开

如果资源路径偏向站点根路径思路，就会导致本地打开时路径解析错误。即使已切换为相对路径，也仍然只是解决了“找得到文件”的一部分问题，并没有自动解决 ESM、预加载、跨文件引用等问题。

### 3.3 本次结论

这次兼容性问题的真实根因可归纳为两层：

1. 资源路径必须相对化，避免站点根路径依赖
2. 构建产物不能继续强依赖默认 ESM + 多文件外链加载模式

仅设置 [`vite.config.ts`](vite.config.ts) 中的相对 `base` 还不够，必须进一步调整输出格式与构建后产物组织方式。

- [`scripts/postbuild-inline-html.mjs`](scripts/postbuild-inline-html.mjs) 负责构建完成后的 HTML 内联处理

## 5. 最终方案概述

本次采用的是“相对路径 + IIFE 输出 + 构建后内联 HTML”的组合方案。

### 5.1 方案组成

#### 5.1.1 相对路径基础

在 [`vite.config.ts`](vite.config.ts) 与 [`vite.config.js`](vite.config.js) 中配置：

- `base: './'`

作用是让构建产物中的资源引用尽量保持相对路径思路，避免依赖站点根目录。

#### 5.1.2 构建输出改为 IIFE

在 [`vite.config.ts`](vite.config.ts) 与 [`vite.config.js`](vite.config.js) 中配置：

- `build.modulePreload = false`
- `build.cssCodeSplit = false`
- `build.rollupOptions.output.format = 'iife'`
- `build.rollupOptions.output.inlineDynamicImports = true`

该组配置的核心目标是：

- 不再依赖模块预加载
- 尽量减少分拆产物
- 将入口脚本收敛为可直接执行的 IIFE 脚本
- 避免运行时再依赖模块图级联加载

#### 5.1.3 尽量内联静态资源

在 [`vite.config.ts`](vite.config.ts) 与 [`vite.config.js`](vite.config.js) 中配置：

- `build.assetsInlineLimit = 100000000`

这一配置的意图不是无限制鼓励大包，而是当前阶段优先提升 `file://` 场景下的资源自包含程度，尽量减少额外资源请求。

#### 5.1.4 构建后把 HTML 外链脚本与样式内联回入口文件

在 [`package.json`](package.json) 中，当前构建命令为：

- `tsc -b && vite build && node ./scripts/postbuild-inline-html.mjs`

其中最后一步执行 [`scripts/postbuild-inline-html.mjs`](scripts/postbuild-inline-html.mjs)，对构建后的 [`dist/index.html`](dist/index.html) 做二次处理，主要包括：

- 移除 `modulepreload` 标签
- 读取本地 JS 文件并改写为内联 `<script>`
- 读取本地 CSS 文件并改写为内联 `<style>`
- 将脚本插入到 `</body>` 前，确保根节点已存在时再执行

最终目标是让发布入口尽量收敛为单个可直接打开的 [`dist/index.html`](dist/index.html)。

## 6. 为什么选择 IIFE + 构建后内联，而不是继续依赖默认 ESM

这是本次决策的关键。结论先行：

> 因为本次目标不仅是“浏览器能在托管环境加载”，而是“同一份构建产物在 `file://` 下也要稳定可直接打开”。默认 ESM 方案对协议环境更敏感，而 IIFE + 构建后内联可以显著降低这种敏感性。

### 6.1 默认 ESM 的优势

默认 ESM 在常规 Web 部署里有明显优点：

- 更符合现代前端默认生态
- 代码拆分与懒加载更自然
- 浏览器模块图语义更清晰
- 适合长期在线部署与缓存优化

如果项目只考虑：

- 本地开发服务器
- GitHub Pages
- 其他 HTTP 静态托管

那么继续使用默认 ESM 完全合理。

### 6.2 但默认 ESM 与本次目标不完全匹配

本次额外要求是必须兼容 `file://`。在该约束下，默认 ESM 有几个问题：

- 对模块脚本加载环境更敏感
- 经常伴随额外模块依赖文件
- 更容易出现浏览器差异行为
- 一旦继续引入动态导入或额外运行时请求，问题会继续放大

也就是说，默认 ESM 并不是错误方案，而是“不够稳”的方案。

### 6.3 IIFE 的价值

把入口收敛为 IIFE 后：

- 浏览器不需要再以模块图方式执行主入口
- 主脚本可以像传统脚本一样直接执行
- 更适合单 HTML 入口的本地直开目标
- 对 `file://` 协议场景更友好

### 6.4 构建后内联的价值

即使输出 IIFE，如果 [`index.html`](index.html) 仍然引用外部 JS/CSS 文件，本地直开时仍然要依赖额外文件读取与路径解析。构建后内联则进一步把风险前移到构建阶段，换取运行阶段更高的稳定性：

- 构建时处理一次
- 运行时少依赖外链
- 最终分发更接近单文件
- 更适合直接发送、归档、离线打开和简单托管

### 6.5 决策总结

因此，本次没有选择“继续沿用默认 ESM 再做小修小补”，而是直接采用：

- 构建阶段收敛输出结构
- 构建后再做 HTML 内联整理

这是一个以兼容性优先、发布稳健性优先为导向的方案。

## 7. 方案落地细节

### 7.1 [`vite.config.ts`](vite.config.ts) 与 [`vite.config.js`](vite.config.js) 的作用

这两个配置文件当前共同表达了以下构建意图：

- 使用相对 `base`
- 关闭模块预加载
- 关闭 CSS 拆分
- 强制 Rollup 输出为 IIFE
- 内联动态导入
- 提高资源内联阈值

这些配置合在一起，目的是让 Vite 默认偏向“更自包含”的单入口产物，而不是依赖外部模块图。

### 7.2 [`package.json`](package.json) 的作用

[`package.json`](package.json) 中的 `build` 命令把构建过程拆成三段串联：

1. TypeScript 编译检查
2. Vite 正式构建
3. 构建后执行 HTML 内联脚本

这意味着当前的“兼容性修复”并不只存在于 Vite 配置里，而是已经成为构建流水线的一部分。

后续如有人改写 `build` 命令，必须意识到：删除最后一步就等于部分撤销本次修复。

### 7.3 [`scripts/postbuild-inline-html.mjs`](scripts/postbuild-inline-html.mjs) 的作用

该脚本是本次方案能真正闭环的关键。

它当前完成的工作包括：

1. 读取 [`dist/index.html`](dist/index.html)
2. 移除 `modulepreload` 标签
3. 找到本地 JS 外链并读取内容
4. 转换为内联 `<script>`，同时处理 `</script>` 转义
5. 在 `</body>` 前插入内联脚本
6. 找到本地 CSS 外链并读取内容
7. 转换为内联 `<style>`
8. 把处理结果重新写回 [`dist/index.html`](dist/index.html)

从维护角度看，这个脚本本质上是一个“构建产物重写器”。后续只要构建输出 HTML 结构变化，就要优先检查这里是否仍然适配。

## 8. 本次实施结果

本次修复完成后，构建链路的目标状态是：

- 通过 [`package.json`](package.json) 中的 `build` 脚本完成正式构建
- 输出的 [`dist/index.html`](dist/index.html) 尽量收敛为单文件入口
- 在本地 `file://` 场景下可直接打开
- 同时保留 GitHub Pages 的可部署性

换言之，本次不是做了两套完全分裂的发布物，而是在同一条构建链路里，尽量兼容：

- 离线本地直开
- 在线静态托管

这也是为什么本次方案虽然稍微增加了构建复杂度，但降低了交付物使用复杂度。

## 9. 后续开发约束

以下约束是本次修复能长期稳定成立的前提。后续开发必须遵守，否则很容易重新引入 `file://` 兼容性问题。

### 9.1 静态资源路径必须保持相对路径思路

后续无论修改 [`vite.config.ts`](vite.config.ts)、[`vite.config.js`](vite.config.js)、[`index.html`](index.html) 还是资源引用方式，都要坚持相对路径思路。

维护要求：

- 不要把资源路径改回站点根路径依赖
- 不要假设部署环境一定挂载在域名根目录
- 不要新增只适用于 HTTP 根路径的硬编码链接

### 9.2 如引入路由，避免 history 路由

如果后续需要页面切换能力升级，优先选择：

- hash 路由
- 应用内状态切屏

应避免直接依赖 history 路由，因为：

- `file://` 下没有服务端回退能力
- GitHub Pages 对深链回退也不天然友好
- history 路由通常要求托管环境额外配合

当前项目如果只是有限页面切换，优先继续采用应用内状态切屏，只有在确实需要 URL 状态表达时再考虑 hash 方案。

### 9.3 谨慎使用运行时 `fetch`

运行时 `fetch` 在 `file://` 下经常会遇到限制或行为差异。后续如需读取数据，优先考虑：

- 构建期打包进代码
- 静态导入
- 预生成内联数据

不要默认认为本地 JSON、文本或模板文件可以在 `file://` 下被稳定 `fetch`。

### 9.4 谨慎使用动态导入

动态导入会重新引入模块图加载问题。即便当前配置中已有 `inlineDynamicImports`，后续如果代码结构变化、构建策略变化或第三方依赖引入新的拆分点，仍可能破坏本地直开兼容性。

因此后续新增 `import()` 时，应先明确：

- 是否真的有必要做懒加载
- 是否会改变当前单入口收敛效果
- 构建后产物是否仍被内联脚本正确覆盖

### 9.5 谨慎使用 Worker

Worker 往往依赖额外脚本地址、Blob URL 或模块化 worker 构建支持。在 `file://` 场景下，这类能力容易引入额外限制。

如果必须使用，应把它视为一个兼容性专项，而不是普通功能追加。

### 9.6 谨慎使用 WASM

WASM 常涉及额外 `.wasm` 文件加载或特定初始化流程，对 `file://` 也较敏感。任何引入都必须验证：

- 构建后资源如何输出
- 是否能被当前内联方案覆盖
- 本地直开时浏览器是否允许加载

### 9.7 谨慎使用外部字体、音频与其他外链资源

以下资源如果通过外链方式引入，都可能削弱本地直开稳定性：

- 外部字体
- 外部音频
- CDN 图标
- 第三方脚本
- 第三方样式

原则上应优先：

- 本地打包
- 相对路径引用
- 构建后尽量收敛

### 9.8 如果未来包体过大，可考虑区分离线构建与在线部署构建

当前方案是兼容性优先，因此会牺牲一定的产物优雅度与缓存效率，例如：

- 更大的单 HTML 文件
- 较少利用浏览器模块缓存
- 较弱的按需拆分能力

如果未来内容大幅增加、包体明显变大，可以考虑拆分成两套构建策略：

1. 离线构建
   - 目标是 `file://` 可直接打开
   - 偏向 IIFE + 全内联或高内联率

2. 在线部署构建
   - 目标是 GitHub Pages 或其他 HTTP 托管性能更优
   - 可重新启用更适合在线场景的 ESM、多文件拆分与缓存策略

但在正式拆分前，不要随意只为性能优化而回退当前兼容性方案。

## 10. 后续修改 checklist

以下 checklist 用于未来继续调整构建方案、资源组织或页面能力时，快速检查是否破坏了本次兼容性修复。

### 10.1 构建配置检查

- [ ] [`vite.config.ts`](vite.config.ts) 与 [`vite.config.js`](vite.config.js) 仍保持相对 `base` 思路
- [ ] 未误删 IIFE 输出相关配置

- [ ] 未把 CSS 再次拆分为依赖复杂外链的形式

### 10.2 构建命令检查

- [ ] [`package.json`](package.json) 中的 `build` 命令仍包含 [`scripts/postbuild-inline-html.mjs`](scripts/postbuild-inline-html.mjs)
- [ ] 未把构建后内联步骤从正式发布链路中删除
- [ ] 如改了输出目录或 HTML 文件名，已同步更新 [`scripts/postbuild-inline-html.mjs`](scripts/postbuild-inline-html.mjs)

### 10.3 产物结构检查

- [ ] 构建后 [`dist/index.html`](dist/index.html) 仍是主要分发入口
- [ ] 内联脚本插入位置仍正确，页面根节点存在后再执行
- [ ] 内联样式仍能覆盖原 CSS 外链职责
- [ ] 未新增必须依赖网络或运行时额外请求的关键启动资源

### 10.4 功能开发检查

- [ ] 新增页面切换方案未引入 history 路由依赖
- [ ] 新增资源引用仍遵循相对路径思路
- [ ] 新增数据加载未默认依赖运行时 `fetch`
- [ ] 新增懒加载、Worker、WASM 等能力前已评估 `file://` 兼容性
- [ ] 新增第三方库没有强依赖 HTTP 环境才能启动

### 10.5 回归验证检查

- [ ] 执行构建后，GitHub Pages 目标场景仍可用
- [ ] 构建后直接打开 [`dist/index.html`](dist/index.html) 时页面可正常渲染
- [ ] 控制台无关键启动报错
- [ ] 样式、脚本与主要交互均正常

## 11. 维护建议

### 11.1 把本次修复视为发布约束，而不是临时补丁

后续维护时，不应把 [`scripts/postbuild-inline-html.mjs`](scripts/postbuild-inline-html.mjs) 看作一次性的权宜之计，而应视为当前发布模型的一部分。

只要项目仍要求：

- 本地 `file://` 直开
- GitHub Pages 同时可用

那么本次方案就仍然成立。

### 11.2 优先验证构建结果，而不是只看开发服务器效果

Vite 开发服务器运行正常，不代表 [`dist/index.html`](dist/index.html) 在 `file://` 下就一定正常。后续所有涉及以下方面的修改，都应优先关注正式构建产物：

- 构建配置
- 资源引用
- 页面启动链路
- 新引入平台能力

### 11.3 若未来切换方案，必须整体替换而不是局部回退

如果未来决定放弃 `file://` 直开支持，或者决定拆分为离线/在线两套构建策略，应作为独立方案整体调整：

- 明确新的交付目标
- 明确新的构建脚本
- 明确新的验证流程

不要只删除某一项配置或某一步脚本，否则很容易形成“Pages 勉强能跑，但本地直开悄悄失效”的半失控状态。

## 12. 结论

本次修复的核心不是单纯把路径改成相对路径，而是围绕“本地 `file://` 可直接打开，同时保持 GitHub Pages 可用”这个目标，对构建输出格式与发布产物结构做了系统收敛。

最终落地方案为：

- 使用相对路径作为资源引用基础
- 使用 IIFE 降低对默认 ESM 加载环境的依赖
- 通过构建后脚本把 JS/CSS 内联回 [`dist/index.html`](dist/index.html)

后续维护时，只要继续遵守本文中的开发约束与 checklist，就能在继续扩展项目的同时，尽量避免破坏这次已经建立好的本地直开兼容性。
